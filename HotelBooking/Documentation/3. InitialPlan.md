Initial Plan to implement Hotel Booking API (EF Core)

Purpose

Create an HTTP API for a Hotel Booking System using Entity Framework Core following SystemDesign.md and the ticket constraints. The work will be delivered in small, reviewable phases. Each phase is independently functional and includes integration tests. I will wait for review and confirmation after each phase before proceeding.

Design alignment (highlights from Documentation/1. SystemDesign.md)

- Room capacities:
  - Single: 1 person
  - Double: 2 people
  - Deluxe: 3 people
- Hotels have a fixed layout: 6 rooms per hotel (2 singles, 2 doubles, 2 deluxe).
- Bookings are ranges: FromDate (inclusive) and ToDate (exclusive).
- It should not be possible to book a room in the past (today is allowed).
- The schema and endpoints required by SystemDesign.md will be implemented (see Required Endpoints below).

Assumptions and constraints

- Target runtime: .NET 10 (project already targets .NET 10)
- Tests: xUnit + Microsoft.AspNetCore.Mvc.Testing (WebApplicationFactory)
- Integration test DB: SQL Server LocalDB with a dedicated test database (to exercise EF Core behaviour including transactions)
- Dates use DateOnly for domain storage; FromDate is inclusive, ToDate is exclusive.
- No authentication required for endpoints.
- Do not add functionality beyond the ticket and SystemDesign.md.

Required Endpoints (implemented exactly as in SystemDesign.md)

- GET /hotel?name={HotelName} => find hotel by name
- GET /rooms?fromDate={fromDate}&toDate={toDate}&guestCount={guestCount} => find available rooms between two dates for a given number of people
- POST /hotel/{hotelId}/rooms/{roomId}/booking => create booking against a room. Body: { "guestCount": number, "fromDate": "YYYY-MM-DD", "toDate": "YYYY-MM-DD" }
- GET /bookings/{id} => find booking details by id
- POST /database/clear => clear database (test/dev helper)
- POST /database/seed => seed database with test data (test/dev helper)

Schema mapping (from SystemDesign.md)

- Hotel: Id (int, autogenerated), Name (nvarchar(128))
- RoomType: Id (int), Name (varchar(32)), Capacity (tinyint)
- Room: RoomId (int, autogenerated), HotelId (int), TypeId (int)
- Booking: BookingId (int, autogenerated), HotelId (int), RoomId (int), FromDate (inclusive) DateOnly, ToDate (exclusive) DateOnly, GuestCount (tinyint)
- RoomAvailability: Id (int, autogenerated), HotelId (int), RoomId (int), FromDate (inclusive) DateOnly, ToDate (exclusive) DateOnly

Answers to open questions (confirmed)

1. Latest possible date: use 9999-12-31 (31/12/9999) as the latest possible date for RoomAvailability records.
2. Concurrency: concurrency checks should be added. It is acceptable to rely on transaction locks for simplicity rather than adding optimistic rowversion-based concurrency, but safeguards must be implemented (see Phase 5).
3. Seed behaviour: create 3 hotels by default when seeding test/dev data.

Phases

Phase 1 — Domain model, EF Core DbContext, and migrations (Reviewable)

Goal
- Implement domain entities and EF Core DbContext following the Schema and assumptions above.
- Configure EF Core so Ids are auto-generated and unique.
- Seed tooling: implement POST /database/seed to create three hotels with 6 rooms (2 of each type) and the RoomType lookup data. Seed should create RoomAvailability records for each room from DateOnly.Today until the configured latest-possible date.
- Implement POST /database/clear to drop/clear data (test helper).

Deliverables
- Domain model classes and DbContext with fluent configuration / data annotations.
- /database/seed and /database/clear endpoints.

Integration tests for Phase 1
- Test POST /database/seed creates a hotel with 6 rooms and RoomType entries.
- Test clear endpoint removes all hotels, rooms, room types, bookings, and availability records.

Review checklist
- Models and DB schema match Documentation/1. SystemDesign.md.
- Seed and clear endpoints behave as expected.
- Migrations build and apply; tests pass against SQL in-memory provider.

Phase 2 — Read endpoints and Swagger (Reviewable)

Goal
- Implement read endpoints required by SystemDesign.md:
  - GET /hotel?name={HotelName}
  - GET /rooms?fromDate={fromDate}&toDate={toDate}&guestCount={guestCount}
  - GET /bookings/{id}
- Enable OpenAPI/Swagger generation and ensure swagger UI is available in development.

Deliverables
- Controllers and DTOs for the read endpoints.
- Program startup wiring for controllers and Swashbuckle.

Integration tests for Phase 2
- Test GET /hotel?name returns expected hotel.
- Test GET /rooms?fromDate=&toDate=&guestCount= returns rooms that have availability records covering [fromDate, toDate) and whose room type capacity >= guestCount.
- Test GET /bookings/{id} returns a 404 for unknown and 200 for an existing booking.
- Test Swagger JSON available at /swagger/v1/swagger.json.

Review checklist
- Endpoints follow the required routes and semantics (inclusive/exclusive dates).
- Swagger present and endpoints documented.

Phase 3 — Booking endpoint, availability updates and validation (Reviewable)

Goal
- Implement POST /hotel/{hotelId}/rooms/{roomId}/bookings with body { guestCount, fromDate, toDate }.
- Enforce business rules:
  - guestCount <= room capacity -> otherwise 400
  - fromDate < toDate and fromDate >= today -> otherwise 400
  - Room cannot be double-booked for any overlapping night: check RoomAvailability coverage -> otherwise 400
- Update RoomAvailability records transactionally when a booking is created. Use inclusive/exclusive semantics when splitting intervals.

Deliverables
- Booking controller action and application service for booking logic.
- Validation and error mapping to 400 responses for business-rule violations.

Integration tests for Phase 3
- Test booking with guestCount > capacity returns 400.
- Test booking with fromDate in the past returns 400.
- Test booking overlapping an existing booking returns 400.
- Test successful booking persists Booking and adjusts RoomAvailability intervals appropriately (e.g., splits coverage).

Review checklist
- Business rules enforced and covered by tests.
- Booking and availability updates are atomic.

Phase 4 — Availability search and edge cases (Reviewable)

Goal
- Implement GET /rooms?fromDate=&toDate=&guestCount= to search by date range and capacity using RoomAvailability records.
- Ensure proper handling of boundary overlaps (FromDate inclusive, ToDate exclusive).

Deliverables
- Availability search controller and DTOs.

Integration tests for Phase 4
- Test searches return expected rooms when none booked.
- Test after bookings, searches exclude rooms that no longer have full coverage for the requested [fromDate, toDate) range.
- Boundary tests for adjacent bookings and exact-match intervals.

Review checklist
- Search behaves correctly for date ranges and capacities.

Testing strategy and CI

- Integration tests will live in tests/HotelBooking.IntegrationTests using xUnit.
- Use SQL Server LocalDB for EF Core in tests, and WebApplicationFactory for HTTP-level integration tests.
- Tests runnable with dotnet test.
