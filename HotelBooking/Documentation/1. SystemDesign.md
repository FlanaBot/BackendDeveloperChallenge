# Assumptions

## Room Sizes
- Single Room only allows 1 Person
- Double Room allows 2 People
- Deluxe Room allows 3 people

## Other Assumptions
- Searching for rooms for parties of more than 3 people is out of scope
- There is no need to record ownership over bookings (ie assign to a user id)
- There is no requirement to cancel a booking
- Hotels have a set layout of rooms (ie. 2 single rooms, 2 double rooms, 2 deluxe rooms)
- There is no requirement to allow the creation of Hotels
- No security concerns over incremental Ids
- No expectation to lazy evaluate room placements to maximise availabilty (ie 'upgrade' users if it allows a longer running booking to be accomodated)
- It should not be possible to book a room that is in the past (but today is fine)

# Schema

## Hotel
- Id int
- Name nvarchar(128)

## Room Type
- Id int
- Name varchar(32)
- Capacity tinyint

## Room
- RoomId int
- HotelId int
- TypeId int

## Booking
- BookingId int
- HotelId int
- RoomId int
- FromDate (Inclusive) Date
- ToDate (Exclusive) Date
- GuestCount tinyint

## Room Availability
- Id int
- RoomId int
- FromDate (Inclusive) Date
- ToDate (Exclusive) Date

# Required Endpoints

## Find a hotel based on its name.
GET /hotels?name={HotelName}

## Find available rooms between two dates for a given number of people.
GET /rooms?fromDate={fromDate}&toDate={toDate}&guestCount={guestCount}

## Book a room.
POST /hotels/{hotelId}/rooms/{roomId}/bookings
Body:
{
	guestCount,
	fromDate,
	toDate
}

## Find booking details based on a booking reference number.
GET /bookings/{id}

## Clear Database
POST /database/clear

## Seed Database with Test Data
POST /database/seed
